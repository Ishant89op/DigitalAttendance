ATTENDX DETAILED PROJECT REPORT
Prepared from current workspace source snapshot

============================================================
1) PROJECT IDENTITY AND PURPOSE
============================================================
Project Name:
- AttendX - Smart Digital Attendance System

Primary Goal:
- Automate classroom attendance using face recognition, while still allowing controlled manual correction, analytics, dispute handling, and administrative data management.

Primary User Roles:
- Student
- Teacher
- Admin
- Classroom Device

Core Problem Solved:
- Replace manual attendance roll calls with camera-based real-time marking.
- Preserve accuracy with manual override and dispute workflows.
- Provide transparent analytics and risk tracking to students, teachers, and administrators.

============================================================
2) HIGH LEVEL ARCHITECTURE
============================================================
Technology Stack:
- Backend API: FastAPI
- Database: PostgreSQL
- Async DB driver: asyncpg
- Recognition: InsightFace + OpenCV + NumPy
- Frontend: Static HTML/CSS/JavaScript dashboards
- Test Stack: pytest + Selenium

Architectural Shape:
- Multi-role web UI (static pages) calls REST API.
- FastAPI router layer delegates to service layer.
- Service layer performs SQL-heavy analytics and workflows.
- Recognition engine runs as a separate process per classroom.
- Recognition process lifecycle is tracked in DB for crash/restart resilience.

Lifecycle (api/server.py):
- Startup:
  - initialize DB pool
  - run schema migrations
  - clean stale recognition sessions
- Runtime:
  - serve /auth, /admin, /lecture, /attendance, /analytics
- Shutdown:
  - stop all recognition processes
  - close DB pool

============================================================
3) RUNTIME ENTRYPOINTS AND COMMANDS
============================================================
Main command router (main.py):
- python main.py db
  - initializes DB pool, runs migrations, closes pool
- python main.py register
  - starts student face registration flow
- python main.py recognize
  - starts recognition for default classroom (CR-2113 or CLASSROOM_ID env)
- python main.py recognize --classroom <ROOM_ID>
  - starts recognition for specific classroom
- python main.py server
  - starts FastAPI on 0.0.0.0:8000
  - optional --reload flag

Environment bootstrap:
- main.py reads .env values itself (without overwriting existing env vars).

- principal_id TEXT

9) ATTENDANCE WORKFLOWS (END TO END)
============================================================
----------------------------------------
9.1 Automated Face Attendance Flow
----------------------------------------
1. Teacher/classroom starts lecture via /lecture/start.
2. lecture_service starts new or resumes same-day session.
3. recognition_manager starts recognizer subprocess for classroom.
4. recognizer detects faces, computes embeddings, matches known matrix.
5. Liveness and cooldown checks are applied.
6. attendance_manager.mark_attendance inserts attendance row idempotently.
7. audit_log receives attendance_marked event.
8. /lecture/{id}/live and /attendance/list/{id} update dashboards.

----------------------------------------
9.2 Teacher Manual Attendance Flow
----------------------------------------
1. Teacher opens Manual Attendance section.
2. Teacher selects course and previous lecture.
3. UI loads absent/all student list.
4. Teacher chooses present/absent action.
5. /attendance/override triggers manual_override.
6. attendance row inserted or deleted.
7. audit_log receives manual_override event.

----------------------------------------
9.3 Student Dispute Flow
----------------------------------------
1. Student opens history and identifies lecture.
2. Student submits dispute via /attendance/disputes.
3. dispute row stored as open.
4. Teacher/admin lists queue via /attendance/disputes.
5. Reviewer resolves via /attendance/disputes/{id}/resolve.
6. If approved and lecture exists, attendance may be marked automatically.
7. dispute status and audit log are updated.

----------------------------------------
9.4 Defaulter Reminder Flow
----------------------------------------
1. System computes low attendance alerts.
2. Admin triggers /analytics/admin/reminders/generate.
3. reminder rows inserted/upserted weekly.
4. Admin views queue via /analytics/admin/reminders.
5. Admin marks sent via /analytics/admin/reminders/mark-sent.

----------------------------------------
9.5 Export Flow
----------------------------------------
Teacher export:
- route: /analytics/teacher/{teacher_id}/courses/{course_id}/export
- formats: excel or pdf
- optional from_date/to_date filtering

Admin export:
- route: /analytics/admin/export
- filters: course_id, department, semester, from_date, to_date
- formats: excel or pdf

11) RECOGNITION AND BIOMETRIC DETAILS
============================================================
Model and Embeddings:
- InsightFace model pack: buffalo_l
- embedding dimension: 512

Matching:
- cosine similarity
- configurable threshold and winner margin
- strict mode flags available via env

Liveness:
- optional motion-based center-shift check
- requires minimum pixel motion before accepting mark

Anti-duplicate controls:
- recognition buffer (multiple frames)
- per-student cooldown window
- DB unique constraint on (student_id, lecture_id)

Process control:
- API tries to start recognizer process
- process PID tracked in recognition_sessions
- stale/dead PID cleanup on startup and health checks
- manual command fallback for Windows behavior

Registration:
- captures configurable sample count
- averages embeddings and normalizes vector
- stores bytes in students.face_encoding
